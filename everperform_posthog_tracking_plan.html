Metrics Framework

Structured using the PM metrics-dashboard framework: North Star → Input Metrics → Health Metrics → Counter-Metrics → Alert Thresholds

All metrics are org-filterable. Every event carries org_id as a super property + PostHog Group Analytics. Any KPI, funnel, or dashboard below can be scoped to a single org, a segment of orgs (by tier, size band), or all orgs.
⭐ NORTH STAR: Weekly Active Managers (WAM)
Unique managers with ≥1 meaningful action per week
Current baseline: ~22% · Target: 50%

├── INPUT: Pulse Response Rate → driven by employee engagement + reminder cadence
├── INPUT: AI Recommendation Accept Rate → driven by relevance of AI outputs
├── INPUT: Ticket Resolution Rate → driven by inbox workflow + assignment
├── INPUT: Feature Toggle Adoption → driven by onboarding + CS enablement
└── INPUT: ELI Chat Adoption → driven by AI coaching value + discoverability

├── HEALTH: AI Pipeline Success Rate → should stay >98%
├── HEALTH: Pulse Email Delivery Rate → should stay >99%
├── HEALTH: API Error Rate → should stay <1%
└── HEALTH: Manager Inactivity Rate → should stay <20%

├── COUNTER: Session Duration → if WAM rises but session time drops, engagement is shallow
└── COUNTER: Ticket Stall Rate → if resolution rate rises but stalls increase, tickets are being closed without action

⭐ North Star

Weekly Active Managers (WAM)

unique managers with ≥1 action (view report, move ticket, accept recommendation, use ELI) per week
Baseline: ~22% · Target: 50%
Why: Managers taking action = the core value loop working. Captures the "push, don't pull" vision.

📊 Input Metrics (drive the North Star)

Pulse Response Rate

pulse_response_submitted / pulse_sent × 100
Target: >75%
No responses = no data = no AI summaries = no tickets = nothing for managers to act on

AI Recommendation Accept Rate

ai_recommendation_accepted / (accepted + dismissed) × 100
Target: >40%
Accepted = child ticket created = actionable AI output. The highest-leverage conversion in the product.

Ticket Resolution Rate

issue_resolved / issue_created × 100 (trailing 30 days)
Target: >70%
Tickets resolved = managers completing the action loop.

Feature Toggle Adoption (%)

orgs with toggle ON / total active orgs (per toggle)
auto_create_tickets: >60% · ai_summaries: >50% · recommendations: >40%
Toggles OFF = features invisible. This gates the entire Golden Flow.

ELI Chat Adoption

unique users with eli_chat_opened / MAU × 100
Target: >30%
ELI = coaching differentiator. Low adoption = discoverability problem.

🛡️ Health Metrics (should stay stable)

AI Pipeline Success Rate

ai_summary_generated(success) / ai_summary_generation_started × 100
Target: >98%
If AI breaks, the entire value chain stops.

Pulse Email Delivery Rate

pulse_email_delivered / pulse_sent × 100
Target: >99%
Emails not delivered = no survey = no data.

API Error Rate

api_error / total API calls × 100
Target: <1%
Backend reliability.

Manager Inactivity Rate

managers inactive >14 days / total managers × 100
Alert: >20%
Silent churn signal — managers disengaging.

⚠️ Counter-Metrics (prevent gaming)

Session Duration

median(user_logged_out.session_duration_s)
Watch: if WAM rises but session time drops below 2 min
Prevents: empty logins counting as "active" without real engagement.

Ticket Stall Rate

issue_stalled / total open issues × 100
Watch: if resolution rate rises but stall rate also rises
Prevents: tickets being closed without meaningful action (fake resolution).

🔬 Key Funnels (PostHog Path Exploration)

Survey Funnel (Employee)

email_delivered → opened → clicked → page_viewed → started → question_answered(×N) → submitted
Identify: which question causes biggest drop-off

AI Value Chain (Manager)

ai_summary_generated → viewed → recommendations_viewed → expanded → accepted/dismissed
Identify: where AI value leaks (generated but never viewed?)

Ticket Lifecycle (Manager)

created → entered_todo → in_progress → review → resolved
Identify: which status transition has longest dwell time

Golden Flow End-to-End (Admin→Manager)

pulse_created → pulse_sent → pulse_closed → ai_summary_generated → issue_created → issue_viewed → issue_resolved
Identify: where the full value chain breaks

Daily

Glance at North Star + health metrics. Check for red alerts.

Super Properties

Auto-attached to every event via posthog.register(). Enables segmentation without repeating properties.

Every metric is filterable by Org. Because org_id is a super property, it's automatically on every single event. Combined with PostHog Group Analytics (organisation), you can filter any insight, funnel, path exploration, or dashboard by organisation — no extra work needed. Just add a filter org_id = X or use the Group Analytics breakdown.
source
"product" | "website" — same PostHog project, differentiated by source
user_role
"admin" | "manager" | "employee"
org_id
Organisation ID — PostHog Groups for org-level analytics
org_name
Human-readable org name
org_tier
"standard" | "pro" | "pro_plus" | "enterprise"
org_size_band
"small" (<50) | "medium" (50-200) | "large" (200+)
team_id
User's primary team
environment
"production" | "staging"
app_version
Release version for regression tracking
is_demo_account
Exclude demo/test accounts

import posthog from 'posthog-js' if (typeof window !== 'undefined') { posthog.init('phc_YOUR_KEY', { api_host: 'https://eu.posthog.com', capture_pageview: true, capture_pageleave: true, autocapture: true, session_recording: { maskAllInputs: true, maskTextContent: true }, person_profiles: 'identified_only', }) posthog.register({ source: 'product', environment: process.env.NEXT_PUBLIC_APP_ENV }) }

Privacy & Compliance

What we track, what we never track, and technical safeguards.

✅ DO Track

  • User ID, email, name
  • Role and permissions
  • Organisation ID and name
  • Subscription tier
  • Feature usage events
  • Timing and durations
  • Counts and rates
  • Error types and codes

❌ NEVER Track

  • Survey response text (free text)
  • AI summary content
  • ELI chat message content
  • Ticket descriptions/comments
  • Individual score values tied to people
  • Employee names in properties
  • Passwords, tokens, credentials
  • IP addresses (disable in PostHog)

🛡️ Technical Safeguards

posthog.init('key', { sanitize_properties: (props) => { delete props['survey_response_text'] delete props['ai_summary_content'] delete props['chat_message_content'] return props } }) // Per-org opt-out if (org.analytics_opt_out) posthog.opt_out_capturing() // PostHog settings: Discard IP, EU instance, 12-month retention
